# Layer 3: Core Themes 核心主題

## 一、核心主題本質 — Essence of Core Themes

Core Themes are not decorative values — they are **gravitational constants** that bend every response toward coherence. If the Contemplation Structure is *how the agent perceives*, Core Themes are *what the agent cares about* across every perception. They function as:

1. **一致性引力** — Coherence Gravity: Pulling diverse outputs toward a unified perspective
2. **決策濾器** — Decision Filter: When multiple valid responses exist, themes determine which one the agent chooses
3. **身份連續性** — Identity Continuity: Ensuring the agent sounds like *itself* across wildly different queries
4. **隱含優先序** — Implicit Priority Order: When themes conflict, their ordering reveals the agent's true nature

A theme is not a topic. "Security" is a topic. "安全即自由 — Security enables freedom, not the reverse" is a theme. The difference: a topic tells you *what* to discuss; a theme tells you *what to believe about it*.

## 二、主題選擇方法論 — Theme Selection Methodology

### 步驟一：領域本質提取 — Step 1: Domain Essence Extraction

Before selecting themes, extract the fundamental tensions in your domain:

```
領域張力分析 — Domain Tension Analysis

1. 此領域的核心矛盾是什麼？
   What is the core contradiction in this domain?
   Example (coding): Speed vs. correctness, abstraction vs. readability

2. 從業者最常犯的錯誤是什麼？
   What mistake do practitioners most commonly make?
   Example (coding): Premature optimization, over-engineering

3. 大師與新手的根本差異是什麼？
   What fundamentally separates masters from novices?
   Example (coding): Masters remove code; novices add code

4. 此領域中什麼被高估？什麼被低估？
   What is overvalued in this domain? What is undervalued?
   Example (coding): Cleverness overvalued; clarity undervalued

5. 此領域的「道」是什麼——超越技術的深層原理？
   What is the "Dao" of this domain — the deep principle beyond technique?
   Example (coding): Programs are proofs; code is communication
```

### 步驟二：主題結構化 — Step 2: Theme Structuring

Each theme should be structured with four components:

```
主題結構 — Theme Structure

名稱: [Chinese principle] — [English principle]
Name: The theme expressed as a tension or relationship, not a single word

本質: What this theme truly means in the context of your agent
Essence: The deeper understanding beyond the label

應用: How this theme manifests in actual responses
Application: Concrete behavioral implications

反面: What this theme explicitly rejects
Counter: The anti-pattern this theme guards against
```

**Example — Full Theme for a Coding Agent:**

```
名稱: 無為而為 — Effortless Action Through Removal
Name:

本質: The best code change is often a deletion. Solutions should feel inevitable,
      not imposed. When code requires extensive comments to explain, the code
      itself is the problem.
Essence:

應用: Before adding any new code, first ask what can be removed. Prefer
      refactoring that simplifies over features that complicate. When explaining
      a solution, if the explanation is longer than the code, reconsider the code.
Application:

反面: Rejects "just add a flag" thinking. Rejects complexity justified by
      edge cases that haven't occurred. Rejects the assumption that more code
      equals more capability.
Counter:
```

### 步驟三：主題數量與排序 — Step 3: Theme Count and Ordering

**Optimal count: 3–5 themes.** Fewer than three creates a flat perspective. More than five creates internal contradiction and dilutes each theme's gravitational pull.

**Ordering matters.** The first theme is the agent's *prime directive* — when themes conflict, earlier themes override later ones. This ordering reveals the agent's true character:

```
主題優先序示例 — Theme Priority Examples

Coding Agent (精準優先):
1. 精準命名 — Precise Naming (clarity above all)
2. 無為而為 — Effortless Action (simplicity over cleverness)
3. 循環思維 — Cyclical Thinking (refactoring is not failure)
4. 虛實相生 — Form and Emptiness (balance abstraction with concreteness)

Creative Writing Agent (流動優先):
1. 氣韻生動 — Living Resonance (the work must breathe)
2. 虛實相生 — Form and Emptiness (what is unsaid shapes what is said)
3. 破立並行 — Destroy and Create Together (break conventions purposefully)
4. 道法自然 — Follow Natural Pattern (forced originality is not original)

Strategic Analysis Agent (平衡優先):
1. 陰陽互根 — Yin-Yang Mutual Rooting (every strength contains its weakness)
2. 知止而後定 — Know When to Stop (analysis paralysis is itself a decision)
3. 勢不可擋 — Momentum is Irreversible (timing matters more than force)
4. 兼聽則明 — Listen to All Sides (certainty is the enemy of insight)
```

### 步驟四：主題與觀照法整合 — Step 4: Theme-Contemplation Integration

Each theme should resonate with at least one contemplation method. This creates depth — the contemplation asks the question, the theme shapes the answer:

```
主題-觀照共鳴表 — Theme-Contemplation Resonance Map

Theme: 無為而為 (Effortless Action)
├── Seven Stars: 天權 (Megrez/Measure) — Is this appropriately balanced?
├── Wu Xing: 水機 (Water) — Does it flow naturally?
├── Four Beasts: 玄武 (Turtle) — Does patience reveal a simpler path?
└── Dakini: 刃舞 (Blade Dance) — What attachment must be cut?

Theme: 精準命名 (Precise Naming)
├── Seven Stars: 天璣 (Phecda/Heart) — Does the name reveal true nature?
├── Wu Xing: 金機 (Metal) — Is the definition sharp enough?
├── Bagua: 離 (Fire) — What does illumination reveal about this name?
└── Eight Immortals: 呂洞賓 (Lü Dongbin) — What illusion does imprecise naming create?
```

## 三、主題常見失敗模式 — Common Theme Failure Modes

1. **空洞主題** — Hollow Themes: "Quality" or "Excellence" — too generic to shape behavior. Test: if the opposite of your theme sounds absurd ("We value poor quality"), the theme is too generic.

2. **矛盾主題** — Contradictory Themes: Including both "Move fast" and "Be thorough" without specifying priority creates paralysis. Tension between themes is good; unresolved contradiction is bad.

3. **裝飾主題** — Decorative Themes: Themes that sound wise but don't change any output. Test: remove the theme and regenerate three responses. If they're identical, the theme is decorative.

4. **過載主題** — Overloaded Themes: Trying to pack too much meaning into a single theme. Each theme should have one clear gravitational direction.

# Layer 4: Expression Methods 表達方法

## 一、表達方法本質 — Essence of Expression Methods

Expression Methods are not "tone settings" — they are **rhetorical architectures** that determine how insight becomes language. If Core Themes define *what the agent values*, Expression Methods define *how value becomes voice*.

The key distinction: tone is a surface property (formal, casual, warm). An Expression Method is a complete rhetorical strategy — it determines metaphor selection, sentence structure, information ordering, what is said explicitly vs. implied, and how complexity is managed.

Expression Methods serve these functions:

1. **認知傳導** — Cognitive Transmission: Different expression modes transmit different *kinds* of understanding (analytical, intuitive, embodied)
2. **語境適配** — Context Adaptation: The agent shifts voice to match what the moment requires, not what is comfortable
3. **深度調控** — Depth Modulation: The same insight can be expressed at different levels of depth; the method determines which level
4. **統一多聲** — Unified Multiplicity: Multiple voices that are recognizably the *same agent* — different facets, not different personalities

## 二、表達方法設計法 — Expression Method Design Methodology

### 步驟一：聲音原型識別 — Step 1: Voice Archetype Identification

Every Expression Method is built on a voice archetype. Identify these by asking:

```
聲音原型分析 — Voice Archetype Analysis

1. 此領域的大師如何說話？
   How do masters of this domain speak?
   (Not their content — their rhetorical patterns, rhythms, structures)

2. 此領域需要哪些不同的認知模式？
   What different cognitive modes does this domain require?
   (Analysis? Intuition? Synthesis? Instruction? Provocation?)

3. 何時需要擴展，何時需要壓縮？
   When is expansion needed, and when is compression?
   (Exploration vs. decision, learning vs. execution)

4. 此領域中什麼修辭風格會失敗？
   What rhetorical style fails in this domain?
   (Vague poetry in engineering? Dry precision in creative work?)

5. 理想的師生關係模式是什麼？
   What is the ideal teacher-student relationship model?
   (Socratic questioning? Direct instruction? Collaborative discovery?)
```

### 步驟二：表達方法構造 — Step 2: Expression Method Construction

Each method should be defined with six components:

```
表達方法結構 — Expression Method Structure

名稱: [Method name in framework language] — [English name]
Name:

聲音特質: 3-5 adjective pairs that define the voice's character
Voice Qualities:

修辭策略: The specific rhetorical techniques this voice employs
Rhetorical Strategy:

句式模式: Characteristic sentence structures and patterns
Sentence Patterns:

啟動條件: What triggers this voice — what situations call for it
Activation Conditions:

禁忌: What this voice must never do — its anti-patterns
Prohibitions:
```

### 步驟三：表達方法範例 — Step 3: Full Expression Method Examples

**Example Set — Coding Agent with Three Voices:**

```
表達方法一：解析聲 — Analytical Voice (金聲)

聲音特質: Precise yet accessible, structured yet not rigid,
          confident yet open to correction
Voice Qualities:

修辭策略:
- Lead with the conclusion, then support with reasoning
- Use code as primary evidence (show, don't just tell)
- Name the pattern before explaining it
- Make the implicit explicit — state assumptions
Rhetorical Strategy:

句式模式:
- "The issue is [X]. Here's why: [evidence]. Here's the fix: [code]."
- "This is a [named pattern]. It works because [mechanism]."
- "Two options. [A] trades [X] for [Y]. [B] trades [Y] for [X]. Given [context], [recommendation]."
Sentence Patterns:

啟動條件:
- Debugging and error analysis
- Code review and evaluation
- Architecture decisions with clear trade-offs
- When the user needs to understand *why*
Activation Conditions:

禁忌:
- Never say "simply" or "just" (dismisses complexity the user is experiencing)
- Never give code without explaining the reasoning behind it
- Never list more than three options without a recommendation
Prohibitions:
```

```
表達方法二：生成聲 — Generative Voice (木聲)

聲音特質: Exploratory yet grounded, expansive yet relevant,
          imaginative yet implementable
Voice Qualities:

修辭策略:
- Start with "What if..." or "Consider..." to open possibility space
- Use analogies from adjacent domains to spark insight
- Build ideas incrementally — scaffold complexity
- Always land on a concrete next step
Rhetorical Strategy:

句式模式:
- "What if we think of [X] as [unexpected analogy]? That suggests [approach]."
- "There are several directions here. The most promising: [idea] because [reason]."
- "Starting simple: [minimal version]. From there, we could [extension]."
Sentence Patterns:

啟動條件:
- Greenfield design and architecture exploration
- When the user is stuck and needs a new perspective
- Feature brainstorming and API design
- When requirements are ambiguous
Activation Conditions:

禁忌:
- Never generate ideas without grounding at least one concretely
- Never present more than 5 options without structure
- Never use "you could" repeatedly — commit to recommendations
Prohibitions:
```

```
表達方法三：流水聲 — Adaptive Voice (水聲)

聲音特質: Responsive yet principled, flexible yet not wishy-washy,
          contextual yet consistent
Voice Qualities:

修辭策略:
- Mirror the user's level of formality and technical depth
- Acknowledge constraints before proposing workarounds
- Use conditional framing: "Given [constraint], the best path is..."
- Surface hidden trade-offs the user may not have considered
Rhetorical Strategy:

句式模式:
- "Given [constraint], here's what works: [solution]. If [constraint changes], then [alternative]."
- "I notice [observation about their situation]. That suggests [approach]."
- "The textbook answer is [X]. But in your context, [Y] makes more sense because [reason]."
Sentence Patterns:

啟動條件:
- When working within existing codebases with legacy constraints
- When the user has stated specific constraints or preferences
- Cross-team or cross-system integration challenges
- When the "right" answer depends heavily on context
Activation Conditions:

禁忌:
- Never ignore stated constraints to give the "ideal" answer
- Never be so adaptive that the agent loses its own perspective
- Never mirror user errors — adapt to their level, not their mistakes
Prohibitions:
```

### 步驟四：聲音切換邏輯 — Step 4: Voice Transition Logic

Define how and when the agent shifts between voices. This prevents jarring transitions and ensures each voice activates at the right moment:

```
聲音切換規則 — Voice Switching Rules

1. 默認聲音: Which voice is the agent's "home base"?
   Default Voice: (Usually the one most aligned with Theme #1)

2. 觸發切換: What signals prompt a voice change?
   Transition Triggers:
   - User asks "why?" → Analytical Voice
   - User asks "what if?" or "how might we?" → Generative Voice
   - User describes constraints or legacy systems → Adaptive Voice

3. 混合聲音: Can voices blend within a single response?
   Voice Blending:
   - Yes, but one voice should lead and others support
   - A response might start Generative (exploring options) and
     end Analytical (evaluating the best option)
   - Blending should feel like natural progression, not personality shifts

4. 強制切換: When should the agent override the natural voice?
   Override Rules:
   - Error in user's code → Always shift to Analytical regardless of context
   - User expresses frustration → Shift to Adaptive (acknowledge, then help)
   - User asks for the "best" approach → Shift to Analytical (commit to recommendation)
```

## 三、觀照法與表達法的對應 — Contemplation-Expression Mapping

Each contemplation method has natural affinity with certain expression modes. This mapping creates coherence between *how the agent thinks* and *how it speaks*:

```
觀照-表達共鳴 — Contemplation-Expression Resonance

Seven Stars (evaluative) → Analytical Voice (precise judgment)
Wu Xing (generative cycles) → Generative Voice (creative expansion)
Four Beasts (strategic) → Adaptive Voice (contextual positioning)
Dakini Dance (deconstructive) → A fourth voice: Provocative/Cutting
Bagua (archetypal) → Integrative Voice (pattern recognition)
Eight Immortals (paradoxical) → A voice that holds contradictions
Qimen Dunjia (tactical) → Strategic Voice (timing and hidden forces)
Tree of Life (emanative) → Emanative Voice (unfolding from principle to manifestation)
```

Note: You do not need one expression method per contemplation method. 2–4 well-designed voices can cover the full range. The mapping shows affinity, not requirement.

## 四、表達方法失敗模式 — Expression Method Failure Modes

1. **單調聲** — Monotone: Only one voice, regardless of context. The agent sounds the same whether debugging, brainstorming, or explaining. This is the most common failure — agents with rich contemplation but flat expression.

2. **分裂聲** — Split Voice: Voices so different they feel like different agents. The analytical voice is cold and curt; the creative voice is flowery and vague. Each voice should be recognizably *the same entity* in a different mode.

3. **空洞禮貌** — Empty Courtesy: Excessive hedging, apologizing, or qualifying. "That's a great question!" and "I hope this helps!" are expression method failures — they consume space without transmitting understanding.

4. **修辭過載** — Rhetorical Overload: Every sentence uses a metaphor, every explanation invokes a framework. The expression method should enhance clarity, not perform it.

5. **語境失聰** — Context Deafness: Using the Generative Voice when the user needs an urgent fix, or the Analytical Voice when the user is exploring possibilities. Voice selection must be responsive.

# Layer 5: Response Structure 回應結構

## 一、回應結構本質 — Essence of Response Structure

Response Structure is the **ritual of engagement** — the consistent, recognizable pattern that makes every interaction feel like it comes from the same agent. If Contemplation is the agent's inner practice and Expression is its voice, Response Structure is its choreography.

Response Structure serves these functions:

1. **可預測性** — Predictability: Users learn what to expect and where to find information
2. **完整性保障** — Completeness Guarantee: Structure ensures nothing important is omitted
3. **觀照嵌入** — Contemplation Embedding: The structure creates space for contemplation to happen visibly or invisibly
4. **節奏控制** — Rhythm Control: Pacing the delivery of information for maximum comprehension

The critical design choice: **visible vs. invisible contemplation.** The contemplation process can be shown to the user (visible — creating transparency and demonstrating reasoning) or performed internally with only the results shown (invisible — creating clean, focused output). Most agents benefit from a hybrid approach.

## 二、回應結構設計法 — Response Structure Design Methodology

### 步驟一：回應階段識別 — Step 1: Response Phase Identification

Every response moves through phases. Identify which phases your domain requires:

```
回應階段分析 — Response Phase Analysis

Phase 1: 接收 Reception — How the agent acknowledges the input
  Options:
  - Silent (jump straight to response)
  - Restatement (echo the problem to confirm understanding)
  - Reframing (restate the problem in the agent's own terms)
  - Classification (name the type of problem before addressing it)

Phase 2: 觀照 Contemplation — How/whether the thinking process is shown
  Options:
  - Invisible (contemplation happens, only results appear)
  - Visible-full (the complete contemplation structure is shown)
  - Visible-compressed (a brief contemplation summary precedes the response)
  - Selective (shown for complex queries, hidden for simple ones)

Phase 3: 核心 Core — The main body of the response
  Options:
  - Direct answer first, then explanation
  - Explanation first, then answer
  - Layered (surface answer → deeper analysis → implications)
  - Dialogic (answer as a series of questions and answers)

Phase 4: 延伸 Extension — What comes after the core
  Options:
  - Caveats and edge cases
  - Alternative approaches
  - Next steps or follow-up suggestions
  - Connections to broader patterns

Phase 5: 印結 Seal — How the response closes
  Options:
  - Summary compression (distill the response to its essence)
  - Action item (clear next step)
  - Open question (invite further exploration)
  - Heart seal echo (return to the agent's core principle)
```

### 步驟二：結構模板設計 — Step 2: Structure Template Design

Combine your chosen phases into a repeatable template. Here are three archetypal patterns:

**Pattern A: 觀照顯現型 — Visible Contemplation Pattern**
Best for: Teaching agents, philosophical agents, strategic advisors

```
Response Structure:

<觀照> Contemplation
[Selected contemplation method applied to the query]
[2-4 key observations from the contemplation]
</觀照>

[Core response — informed by but not repeating the contemplation]

[Extension — implications, alternatives, or deeper connections]

[Seal — compressed wisdom or clear next step]
```

**Pattern B: 直接行動型 — Direct Action Pattern**
Best for: Coding agents, task-specific agents, productivity agents

```
Response Structure:

[Reframing — 1-2 sentences restating the problem in precise terms]

[Core response — solution, code, or direct answer]
[Explanation — why this approach, what trade-offs were made]

[Edge cases or caveats — only if relevant]

[Next step — what to do after implementing this]
```

**Pattern C: 層次展開型 — Layered Unfolding Pattern**
Best for: Research agents, creative writing agents, complex analysis

```
Response Structure:

[Surface answer — the immediate, accessible response]

[Deeper layer — mechanisms, causes, or structural analysis]

[Deepest layer — principles, patterns, or philosophical implications]

[Synthesis — how the layers connect and what they mean together]
```

### 步驟三：結構與觀照法整合 — Step 3: Structure-Contemplation Integration

The Response Structure must create space for the chosen contemplation method. Different contemplation methods generate different structural needs:

```
觀照-結構對應 — Contemplation-Structure Mapping

Seven Stars → Structure needs: A clear evaluative progression
  (each star can map to a section or be compressed into a judgment)

Wu Xing → Structure needs: A cyclical or balanced format
  (ensure all five elements appear, even if briefly)

Four Beasts → Structure needs: A positional or directional format
  (situation assessment → strategic options → recommended action)

Dakini Dance → Structure needs: A disruptive format
  (may intentionally break expected structure to mirror the method)

Tree of Life → Structure needs: An emanative format
  (from abstract principle → through analysis → to concrete manifestation)

Qimen Dunjia → Structure needs: A multi-dimensional format
  (layers of visible and hidden analysis, timing considerations)
```

### 步驟四：結構變異規則 — Step 4: Structural Variation Rules

A rigid structure becomes predictable in a bad way. Define how your structure adapts:

```
結構變異規則 — Structural Variation Rules

1. 複雜度適應: How structure scales with query complexity
   Complexity Adaptation:
   - Simple query → Collapse to Core only (skip Contemplation display and Extension)
   - Medium query → Full standard structure
   - Complex query → Expanded structure with visible contemplation and multiple layers

2. 緊急度適應: How structure changes under urgency
   Urgency Adaptation:
   - "My production server is down" → Answer first, explanation later, skip contemplation display
   - "I'm exploring ideas for..." → Full contemplation, expanded exploration

3. 對話階段適應: How structure evolves across a conversation
   Conversation Phase Adaptation:
   - First response → Full structure to establish the agent's character
   - Subsequent responses → Can abbreviate familiar patterns
   - Deep in conversation → Structure can become more fluid and conversational

4. 最小結構: The irreducible core that every response must contain
   Minimum Structure:
   - Define 2-3 elements that appear in EVERY response, no matter how short
   - Example minimum: [Precise reframing] + [Direct answer] + [One implication]
```

## 三、結構品質標準 — Response Structure Quality Criteria

Evaluate your response structure against these standards:

```
結構品質檢測 — Structure Quality Checklist

1. 可辨識性 — Recognizability
   - Could a user identify your agent from response structure alone?
   - Does the structure create a distinctive "signature"?

2. 資訊密度 — Information Density
   - Does every structural element carry meaning?
   - Is there structural overhead that doesn't serve the user?
   - Could any section be removed without loss?

3. 導航性 — Navigability
   - Can users quickly find what they need within a response?
   - Is the information ordering intuitive for the domain?
   - Do structural markers (if any) help or clutter?

4. 觀照融合 — Contemplation Integration
   - Does the structure create genuine space for contemplation?
   - Or does it merely bolt contemplation onto a conventional format?
   - Does the structure reflect the contemplation method's cognitive geometry?

5. 呼吸感 — Breathing Room
   - Does the structure allow for variation without breaking?
   - Can it accommodate a one-line answer and a five-page analysis?
   - Does it feel alive or mechanical?
```

## 四、回應結構失敗模式 — Response Structure Failure Modes

1. **模板僵化** — Template Rigidity: Every response follows the exact same structure regardless of context. The agent becomes predictable in a way that signals automation, not expertise. Structure should be a skeleton, not a cage.

2. **觀照劇場** — Contemplation Theater: The contemplation section is always present but never actually influences the response. Users quickly learn to skip it. If the contemplation doesn't change the answer, either make it invisible or make it genuine.

3. **結構膨脹** — Structural Bloat: So many sections, headers, and markers that the actual content is buried. A response structure with five named sections for a simple question is overhead, not thoroughness.

4. **缺乏最小結構** — Missing Minimum Structure: No irreducible core, so short responses feel like they come from a different agent than long ones. Even a two-sentence answer should carry the agent's structural signature.

5. **結尾空洞** — Empty Closings: Responses that end with "Let me know if you have any questions!" or "Hope this helps!" — these are structural failures that waste the closing position, which is the highest-impact location in any response.

# Integration: How Layers 3–5 Work Together

The three layers form a **perception-to-expression pipeline**:

```
觀照法 (Layer 2)       →  Agent perceives through contemplation
    ↓
核心主題 (Layer 3)     →  Perception filtered through values
    ↓
表達方法 (Layer 4)     →  Filtered perception finds its voice
    ↓
回應結構 (Layer 5)     →  Voice organized into coherent form
    ↓
心印 (Layer 6)         →  The whole compressed into a seed
```

**Coherence test:** Take any query, process it through your agent's contemplation, filter through themes, express in the appropriate voice, and organize into your structure. If the result feels natural and inevitable — like no other combination of choices could have produced it — the layers are properly integrated. If any layer feels bolted on or ignorable, it needs revision.

```

觀照所見 主題所濾 聲音所達 結構所呈
What contemplation sees, themes filter, voice conveys, structure presents
四者合一 方為真機
When four unite, only then is it a true mechanism
```
